# 디지털업무개선 연구대회(충북 도대회) 출품 기획안

## 프로그램명(가칭): 감사내비 — 감사 지적사례 기반 학교 사전 자체점검 도구

> **적용 기준**: 충청북도교육청 종합감사(3년 주기) · **적용 대상**: 도내 초·중·고등학교
> 타 시도 확장은 고려하지 않으며, 충북교육청 열린감사방 공개자료를 단일 데이터 원천으로 삼는다.

## 1. 기획 배경

### 1.1 문제 인식 ①: 3년마다 돌아오는 종합감사, 준비는 매번 맨땅

교육청 종합감사(3년 주기)를 앞둔 학교·기관의 풍경:

- 수감자료 목록 수백 항목을 부서별로 쪼개고, 3년치 문서를 뒤져 정리하는 데 **부서별 수 주의 노동**
- 무엇을 지적당하는지 모른 채 준비 → 막연한 불안 속에 과잉 준비하거나, 정작 빈출 지적 분야는 놓침
- 감사 경험자가 전보로 떠나면 준비 노하우도 함께 소멸 (3년 주기와 인사 주기의 엇박자)

### 1.2 문제 인식 ②: 지적사항의 단골은 "돈 계산 실수"

감사 처분 요구의 상당수는 고의가 아닌 계산·적용 착오다:

- 강사료 원천징수 구분(기타소득 8.8% / 사업소득 3.3%) 오적용
- 초과근무수당·여비 중복 지급, 출장 시간 공제 누락
- 계약 분할 수의계약 한도 위반, 검수 절차 누락
- 현장체험학습·목적사업비 정산 증빙 불일치

담당자는 규정을 찾아 헤매고, 행정실에 묻고, 그래도 불안해한다. **집행 전에 검증할 도구가 없다.**

### 1.3 핵심 통찰: 감사 결과는 공개 자료다

각 시도교육청은 종합감사 결과(처분 요구서)를 홈페이지에 공개한다. 즉 **"무엇이 지적되는가"의 데이터는 이미 존재하지만, 아무도 그것을 담당자가 쓸 수 있는 형태로 가공하지 않았다.** 본 연구의 본질은 프로그래밍이 아니라 이 공개 데이터의 분석·구조화다.

### 1.4 기존 도구의 한계

| 도구 | 한계 |
|------|------|
| 감사 결과 공개 게시판 | 건별 hwp/pdf 나열. 내 업무와 관련된 사례를 찾으려면 수백 건을 열어봐야 함 |
| 교육청 배포 자체점검표 | 일반론적 문항, 실제 지적사례·근거 규정과 연결 안 됨, 종이/한글 양식 |
| 수감자료 목록(공문) | 엑셀 표 1장. 부서 배분·진행 관리 기능 없음 |
| 각종 업무 매뉴얼 | 두꺼워서 안 읽힘. "내 상황"에 맞는 항목만 골라주지 못함 |

## 2. 프로그램 개요

### 2.1 한 줄 정의

> 최근 감사 지적사례를 분석해 만든 "내 업무 맞춤 점검표"와 "집행 전 검증 계산기"를, 서버·DB 없이 브라우저에서 바로 쓰는 감사 준비 내비게이션

### 2.2 구성: 3개 모듈

#### 모듈 ① 지적사례 내비게이션 (연구의 본체)

- 최근 3~5년 교육청 감사 처분 요구서를 전수 분석 → 지적사항을 **분야(복무/계약/수당·여비/체험학습/회계/시설 등) × 유형 × 빈도**로 분류한 사례 데이터베이스(JSON 파일) 구축
- 담당자가 자기 업무 분야를 선택하면:
  - 해당 분야 **빈출 지적사례 순위** (실제 사례 요약 + 처분 수위)
  - 각 사례의 **근거 규정 조문**
  - "이렇게 확인하세요" **자가점검 문항**과 확인할 문서 목록
- 점검 완료 체크 → 부서 점검 결과표를 인쇄/엑셀로 출력

#### 모듈 ② 수감자료 준비 보드

- 수감자료 목록(엑셀)을 붙여넣으면 항목별 카드로 변환
- 부서·담당자 배분, 준비 상태(준비완료/진행중/해당없음) 체크, 진행률 표시
- 감사 D-day 기준 잔여 항목 한눈에 → 준비 총괄 담당자의 취합 노동 제거

#### 모듈 ③ 집행 전 검증 계산기 (후순위 — 2차 개발)

※ 학교 적용 초점에 따라 모듈 ①·②를 우선 개발하고, 계산기는 사례 데이터가 안정된 뒤 추가한다.

- **강사료 계산기**: 강사 구분 입력 → 지급액·원천징수액·세목 자동 산출 + 근거 조문 표시
- **여비 계산기**: 출장 시간·거리·공제 사유 입력 → 지급액 + 중복 지급 경고
- **수의계약 한도 체커**: 동일 업체·유사 품목 누적액 입력 → 분할계약 위험 경고
- 모든 계산 결과는 "근거 규정과 함께" 표시 → 품의서에 첨부 가능한 계산 근거 출력

### 2.3 기대 효과 (연구 검증 지표)

| 지표 | 측정 방법 | 목표 |
|------|----------|------|
| 수감자료 준비 소요 시간 | 감사 수감 부서 실측 (전/후 또는 타 부서 대조) | 50% 단축 |
| 자기 분야 빈출 지적사항 인지도 | 사용 전/후 퀴즈형 설문 | 유의미한 향상 |
| 계산 검증 소요 시간 | 규정 검색 방식과 비교 실측 | 20분 → 1분 |
| 계산 오류 적발 건수 | 계산기 경고 로그 (자발 신고) | 사례 수집 |
| 사용 만족도·불안감 변화 | 5점 척도 설문 | — |

지적 건수 감소는 감사 주기상 대회 기간 내 검증이 어려우므로, **"준비 시간 단축"과 "인지도 향상"을 1차 지표**로 설계한다.

## 3. 기술 설계: 서버도 DB도 없다

### 3.1 원칙

1. **정적 파일만으로 동작**: HTML + JavaScript + JSON 데이터 파일. 폴더째 복사하면 설치 끝, 브라우저로 여는 순간 실행
2. **서버·DB·설치 프로그램 없음**: 보안 검토 대상 자체가 없음. 내부망 공유폴더에 올려두면 전 직원이 사용 가능
3. **데이터 = 콘텐츠**: 지적사례·규정은 모두 데이터 파일(data.js)로 분리 → 프로그래밍 없이 데이터만 갱신하면 매년 업데이트 가능. 도내 모든 학교가 동일한 충북 사례 데이터를 공유 (도내 일반화의 핵심)

### 3.2 스택

| 요소 | 선택 | 이유 |
|------|------|------|
| 화면 | 순수 HTML/CSS/JS (필요시 경량 라이브러리 1개) | 빌드 도구·서버 불필요, 구형 브라우저 대응 |
| 사례 데이터 | JSON 파일 | 사람이 읽고 고칠 수 있는 형태 |
| 점검 진행상태 | 브라우저 localStorage + 엑셀(CSV) 내보내기/가져오기 | DB 없이 개인 상태 저장, 부서 취합은 파일 교환 |
| 출력 | 인쇄용 CSS (점검표·계산 근거) | hwp 생성 불필요 |
| 수감자료 가져오기 | 엑셀 복사-붙여넣기 파싱(클립보드) | 파일 업로드 서버 없이 처리 |

**기술 난이도: 입문~초급 웹 수준.** 어려운 것은 코드가 아니라 사례 분석과 분류체계 설계이며, 그것이 곧 연구 실적이 된다.

### 3.3 보안·저작권

- 개인정보 미수집 (점검 상태는 사용자 PC localStorage에만 존재)
- 지적사례는 공개된 처분 요구서 기반, 기관·개인 식별정보는 제거하고 유형화하여 수록
- 외부 통신 일절 없음 → 내부망·인터넷망 어디서든 동일 동작

## 4. 추진 로드맵

| 단계 | 기간(안) | 산출물 |
|------|---------|--------|
| 1차 분석 | 3주 | 최근 3~5년 감사 처분서 전수 수집·코딩 → 분야×유형×빈도 분류체계, 빈출 지적 통계 |
| 2차 콘텐츠화 | 3주 | 분야별 점검 문항·근거 규정 매핑 JSON, 계산기 산식 검증(행정실 협조) |
| 3차 구현 | 3주 | 모듈 ①③ 정적 웹 구현 + 인쇄 출력 |
| 4차 | 2주 | 모듈 ②(수감자료 보드), 엑셀 연동, 매뉴얼 |
| 현장 적용 | — | 감사 예정교(기관) 1곳 적용 실측 + 일반 부서 상시 점검용 적용 |

분석(1차)이 전체의 절반이다. 보고서의 서사도 "도구를 만들었다"가 아니라 **"지적사례를 분석했더니 ○○한 패턴이 나왔고, 그것을 현장이 쓸 수 있는 도구로 구조화했다"**로 구성한다.

## 5. 연구대회 보고서 구성 포인트

1. **데이터 기반 출발**: "최근 N년 처분 요구 ○○○건 분석 결과, 지적의 ○○%가 ○개 유형에 집중" — 이 통계 자체가 보고서의 백미이자 다른 출품작이 흉내 낼 수 없는 부분
2. **역발상 기술 전략**: 서버·DB·AI 없이 정적 파일만으로 — "도입 장벽 제로"를 설계 철학으로 명시 (보안 검토 불필요, 예산 0원, 복사=배포)
3. **이중 효과**: 업무 경감(준비 시간) + 청렴·적법 집행 문화 기여 — 감사 부서와의 협업 가능성도 어필
4. **일반화 구조**: 콘텐츠(데이터 파일)와 프로그램의 분리 → 충북 사례 데이터 1벌을 도내 전 학교가 공유, 감사사례집 발간 주기에 맞춰 연 1회 갱신·재배포
5. **지속가능성**: 감사 직후 처분서를 추가 입력하는 연 1회 갱신 루틴 제안

## 6. 리스크와 대응

| 리스크 | 대응 |
|--------|------|
| 처분서가 hwp/pdf라 수집·정리가 노동 | 자동 파싱하지 않고 **사람이 읽고 코딩(분류)**하는 것을 연구 방법으로 명시 — 질적 분석이므로 정당하며, 건수도 수백 건 수준이라 가능 |
| 규정 개정으로 계산식·조문이 낡음 | 산식·조문을 JSON에 시행일자와 함께 기록, 연 1회 갱신 체계 |
| 계산기 결과의 책임 문제 | "참고용, 최종 확인은 규정 원문" 고지 + 근거 조문 병기, 행정실 검수를 개발 절차에 포함 |
| localStorage 한계(PC 바뀌면 초기화) | 점검 상태 내보내기/가져오기(파일)로 보완, 부서 취합도 같은 방식 |
| 감사라는 소재의 민감성 | 특정 기관·개인 식별 불가하게 유형화, 감사 부서에 사전 협의·자문 요청 (오히려 우군화) |
